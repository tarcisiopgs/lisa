# Stack-Aware Prompt Injection

## Problem

When lifecycle is in `skip` mode (default), agents have no infrastructure services running. Some projects need resources that are typically generated by tools requiring running services (e.g., database migrations, API type generation). Without these resources, agents may fail or produce incorrect implementations.

## Solution

Lisa detects the project's stack tools, resolves infrastructure availability, and injects contextual instructions into the agent prompt — either "run this command" (infra available) or "create these files manually following project conventions" (infra unavailable).

## Design

### 1. Stack Detection (expand `discovery.ts`)

New `discoverStackTools(cwd)` function returns detected tools with contextual metadata.

```typescript
interface StackTool {
  name: string;
  category: "orm" | "api-codegen" | "other";
  language: string;
  configFile: string;
  infraCommand?: string;
  manualHint: string;
}
```

#### Supported tools (initial)

| Tool | Language | Detection | infraCommand | manualHint (summary) |
|------|----------|----------|-------------|----------------------|
| Prisma | TS | `prisma/schema.prisma` | `npx prisma db push` | Create migrations in `prisma/migrations/` |
| Drizzle | TS | `drizzle.config.ts` | `npx drizzle-kit push` | Create SQL migrations in configured folder |
| TypeORM | TS | `data-source.ts` / `ormconfig.*` | `npx typeorm migration:run` | Create migrations in `src/migrations/` |
| Sequelize | TS/JS | `.sequelizerc` / `config/config.json` | `npx sequelize-cli db:migrate` | Create migrations in `migrations/` |
| Rails | Ruby | `Gemfile` + `db/migrate/` | `bundle exec rails db:migrate` | Create migrations in `db/migrate/` with timestamp |
| Django | Python | `manage.py` + `*/migrations/` | `python manage.py migrate` | Create migrations in `<app>/migrations/` |
| Alembic | Python | `alembic.ini` | `alembic upgrade head` | Create migrations in `alembic/versions/` |
| Laravel | PHP | `artisan` + `database/migrations/` | `php artisan migrate` | Create migrations in `database/migrations/` |
| Flyway | Java | `flyway.conf` / `db/migration/` | `flyway migrate` | Create SQL migrations in `db/migration/` |
| Liquibase | Java | `liquibase.properties` / `changelog.*` | `liquibase update` | Create changesets in the changelog |
| Goose | Go | `goose` in `go.mod` + migrations dir | `goose up` | Create SQL/Go migrations in migrations folder |
| golang-migrate | Go | `migrate` in `go.mod` | `migrate up` | Create SQL migrations in configured folder |
| Orval | TS | `orval.config.ts` | `npx orval` | Create TS types based on local OpenAPI spec |
| Kubb | TS | `kubb.config.ts` | `npx kubb generate` | Create TS types based on local OpenAPI spec |
| GraphQL Codegen | TS | `codegen.ts` / `codegen.yml` | `npx graphql-codegen` | Create TS types based on GraphQL schema |
| Protobuf | Multi | `*.proto` + `buf.yaml` / `buf.gen.yaml` | `buf generate` | Create stubs manually following .proto files |

Runtime enrichment: Lisa scans existing project structure (migration folders, last created files) and complements `manualHint` with concrete paths and patterns.

### 2. Infra Status Resolution

```typescript
type InfraStatus = "available" | "unavailable";

function resolveInfraStatus(
  lifecycleMode: LifecycleMode,
  lifecycleResult: { success: boolean }
): InfraStatus
```

| Mode | Result | Status |
|------|--------|--------|
| `auto` | success | `available` |
| `auto` | failure | `unavailable` |
| `validate-only` | success | `available` |
| `validate-only` | failure | `unavailable` |
| `skip` | — | `unavailable` |

Key change: when `auto` or `validate-only` fails, instead of aborting the session, Lisa falls back to manual mode (prompt with manual creation instructions). The session continues.

### 3. Prompt Injection

New function generates a "Resource Generation" block injected into the agent prompt.

**When infra unavailable:**
```
## Resource Generation

The following tools were detected but infrastructure services are not running.
Create resources manually following project conventions:

### Prisma (ORM)
- Do NOT run `npx prisma db push` — no database is available
- Create migration files manually in `prisma/migrations/` following the existing pattern
- Existing migrations: `0001_init/migration.sql`, `0002_add_users/migration.sql`
- Reference `prisma/schema.prisma` for the current schema

### Orval (API Codegen)
- Do NOT run `npx orval` — the API server is not running
- Create TypeScript types manually based on the OpenAPI spec at `openapi.yaml`
- Existing generated types are in `src/api/generated/`
```

**When infra available:**
```
## Resource Generation

Infrastructure services are running. Use the following commands to generate resources:

### Prisma (ORM)
- Run `npx prisma db push` to sync schema with the database
- Run `npx prisma generate` to regenerate the client
```

The block is only injected when `StackTool[]` is non-empty.

### 4. Session Integration

Updated flow in all three session types:

```
discoverInfra() -> discoverStackTools() -> runLifecycle()
-> resolveInfraStatus() -> injectStackInstructions(tools, status)
-> build prompt with instructions -> run agent
```

Changes:
- `worktree-session.ts`, `branch-session.ts`, `multi-repo-session.ts`: remove early return/abort on lifecycle failure. Mark `infraStatus = "unavailable"` and continue.
- Agent prompt receives the Resource Generation block based on status.
- `stopResources()` still called in cleanup normally.
- Multi-repo: detection and injection happen per step (backend may have Prisma, frontend may have Orval).

### 5. Tests

**Unit:**
- `discovery.test.ts`: cases for each new tool detection with filesystem fixtures
- `resolveInfraStatus()`: test mode x result -> status matrix
- `injectStackInstructions()`: test prompt generation for both statuses, with and without existing migrations

**Integration:**
- Session with lifecycle `skip` + Prisma detected -> prompt contains manual instructions
- Session with lifecycle `auto` + success + Prisma -> prompt contains `npx prisma db push`
- Session with lifecycle `auto` + failure + Prisma -> prompt contains manual instructions (fallback)
- Multi-repo: backend with Rails + frontend with Orval -> each step has correct instructions

**Not tested:**
- Whether the agent actually follows the instructions — that's the agent's responsibility, not Lisa's
